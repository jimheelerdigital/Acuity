# Acuity SEO + Visibility Audit — 2026-05-01

**Site:** https://getacuity.io
**Stack:** Next.js 14, Vercel, Supabase
**Auditor:** Claude Code
**Requested by:** Keenan

---

## Executive summary

- **P0 CRITICAL: Domain redirect mismatch.** `getacuity.io` (non-www) 307-redirects to `www.getacuity.io`, but the canonical URLs, sitemap, and all internal metadata reference `getacuity.io`. Google cannot resolve which domain to index. Link equity is being split and partially lost through a temporary (307) redirect.
- **P0 CRITICAL: All /for/ landing pages are noindexed.** Five high-quality angle pages (therapy, founders, sleep, decoded, weekly-report) plus 20+ dynamic persona pages are set to `noindex, nofollow`. These 1,200-2,400 word pages are invisible to search engines — the single largest organic traffic opportunity being left on the table.
- **P0 CRITICAL: Sitemap excludes /for/ pages entirely.** Even after removing noindex, Google won't discover these pages efficiently because the sitemap only lists 17 URLs (homepage, signup, blog listing, pillar page, and 13 blog posts).
- **Quick win: Add llms.txt and AI bot rules.** No llms.txt exists and robots.txt has no specific rules for GPTBot, ClaudeBot, or PerplexityBot. Adding these takes ~30 minutes and opens the door to AI search engine visibility.
- **Quick win: Fix Twitter card type on /for/ pages.** All /for/ pages use `twitter:card = "summary"` (small thumbnail) instead of `"summary_large_image"`. One-line fix per layout file.

**Overall site grade: C+** — The homepage and blog system are reasonably well-optimized (solid structured data, Article schema, canonical URLs). But the redirect mismatch undermines the entire domain's authority, and the noindexed landing pages represent a massive missed opportunity. Fix the three P0s and this becomes a B+.

- **File:** `apps/web/src/components/landing.tsx:1096, 1964`
- **Current state:** Landing page uses raw `<img>` tags instead of `next/image`. Two instances found for the Acuity logo.
- **Issue:** Raw `<img>` tags bypass Next.js image optimization (no automatic WebP/AVIF conversion, no responsive sizes, no lazy loading). For small logos this is minor, but any hero images or phone mockups using `<img>` would be a performance issue. **P2**

#### 4.5 Homepage `force-dynamic`
- **File:** `apps/web/src/app/page.tsx:8`
- **Current state:** `export const dynamic = "force-dynamic"` — the homepage is server-rendered on every request (no caching) because it checks for a session and redirects logged-in users to /home.
- **Issue:** This means the homepage never benefits from Vercel's edge cache. Every visitor gets a cold server render. For the most important page on the site, this hurts Time to First Byte (TTFB). **P1**
- **Recommended fix:** Use middleware to handle the auth redirect instead of forcing the homepage to be dynamic. Or use a static landing page with client-side auth check.

#### 4.6 Client components as pages
- **Current state:** 12 page.tsx files include `"use client"`, including all /for/ pages, /waitlist, and auth pages.
- **Issue:** Client-side rendered pages send more JavaScript to the browser and may have worse LCP. The /for/ pages are content-heavy landing pages that would benefit from being server components. **P2**
- **Priority:** P2
